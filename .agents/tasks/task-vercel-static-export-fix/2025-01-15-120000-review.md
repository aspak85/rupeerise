# Static export for Vercel Hobby tier

The change re-enables `output: "export"` in `next.config.ts` to produce a fully static site, eliminating all serverless functions and staying within Vercel Hobby's limits. The server-side `getPlans()` fetch and ISR `revalidate` export are removed from the homepage. Plans are now fetched client-side in a new `PlansSection` component that renders hardcoded fallback data immediately and swaps in live data after mount.

**Watch for:** The `NEXT_PUBLIC_API_URL` fallback to `localhost:4000` in the production bundle means plan fetching silently fails for every visitor unless the env var is set at build time (confirmed). Vercel rewrites in `vercel.json` don't apply to client-side `fetch()` calls made from the browser, so the component must hit the absolute backend URL directly (confirmed — the component does this correctly, but only if the env var is configured).

## High-level view

The static export config is minimal and correct. No dynamic route segments exist, no server-only APIs are used anywhere in the app, and all pages already use `"use client"` with `useEffect` for data fetching. This change is purely about the homepage.

The client-side fetching gap is error UX — a failed fetch silently keeps fallback data with no indication to the user that prices might be stale. The previous ISR approach retried every 5 minutes; the client-side version never retries after mount.

The `vercel.json` rewrites route `/api/*` to Render at the edge layer, which works with static deployments. However, `PlansSection` fetches from `NEXT_PUBLIC_API_URL` (absolute backend URL) rather than relative `/api/plans`, bypassing the rewrite. This couples the component to the env var being correctly set in Vercel's build environment.

1. **Fallback localhost in production bundle** — `process.env.NEXT_PUBLIC_API_URL || "http://localhost:4000"` is baked at build time. If an admin changes plan pricing via the backend, the fallback data shown during the first render (and during backend outages) will be wrong until a developer manually updates the constant and redeploys. ## PlansSection fetch and the env-var bake-in

`NEXT_PUBLIC_*` variables in Next.js are inlined into the JavaScript bundle at build time. The expression `process.env.NEXT_PUBLIC_API_URL || "http://localhost:4000"` in `PlansSection.tsx` becomes a string literal in the output. If Vercel's build environment doesn't have `NEXT_PUBLIC_API_URL` set, the production bundle will contain `"http://localhost:4000"` — a URL that resolves to nothing in visitors' browsers.

The safer pattern for a static export deployed to Vercel would be to fetch from a relative path like `/api/plans` and let the `vercel.json` rewrite proxy it to Render. This avoids exposing the backend URL in client JS and eliminates the env-var misconfiguration risk. The rewrite infrastructure already exists:

```json
{ "source": "/api/:path*", "destination": "https://rupeerise-api.onrender.com/:path*" }
```

Using it from `PlansSection` would mean changing one line:

```ts
fetch(`/api/plans`, { signal: ctrl.signal })
```

This also makes the component consistent with the rest of the app (login, dashboard, wallet pages all call relative `/api/` paths that go through the rewrite).

## Silent failure and stale fallback

The `.catch(() => {})` in the effect means network errors, CORS failures, and JSON parse errors all result in the user seeing `FALLBACK_PLANS` with no indication that the data is stale. The server-side version at least re-tried every 5 minutes via ISR. The client-side version never retries — once the page loads with fallback data, it stays that way until a full page refresh.

## Static export compatibility — no remaining issues

A scan of the full `apps/web/src/` tree found zero uses of `revalidate`, `getServerSideProps`, `getStaticProps`, `cookies()`, or `headers()`. No dynamic route segments (`[param]` directories) exist. The `Counter` component uses `require("next/dynamic")` with `{ ssr: false }`, which is compatible with static export. The build produces all 27 pages as static HTML.
